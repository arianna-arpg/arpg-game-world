Under the hood

One engine. Everything else is data.

Hollow Wake is built around a single idea: skills, monsters, statuses, passives, items, biomes, and world-events are all plain data entries, composed by one shared engine.

This page is the deep dive for the curious — how the systems interlock, why adding content almost never touches engine code, and how it all stays honest. If you're here to play, the home page is the tour you want; this is the machine room.

The modifier engine

Every number flows through one formula

There is exactly one place a stat is computed. Flat additions, then increased (additive percentages), then more (multiplicative), then a hard override — every modifier in the game, from a passive node to a monster's innate to a unique's downside, is just a line that joins one of these layers. Content authors compose behavior out of tags and modifiers instead of writing new engine code.

1

flat

Raw additions to the base. +35 Maximum Life

2

increased

Additive percentages, summed together. +47% increased melee damage

3

more

Multiplicative — each one its own factor. ×1.25 more

4

override

Replaces the result outright. Keystones live here.

value = override ?? ( (base + Σflat) × (1 + Σincreased) × Π(1 + more) ) // then clamped to the stat's min/max, filtered by tags

Because it's tag-scaled, the same line means different things in different contexts: 10% increased damage [melee] only applies to melee. Tags are how one formula serves a whole game.

Actors

The player, the monsters, and your minions all run the same skills

There is one entity model and one skill pipeline — World.useSkill(). A fireball behaves identically whether a sorcerer casts it, a monster breathes it, or your summoned skeleton throws it. A summoner's minions literally run on the same skill definitions the monsters do. Classes are starting kits, not cages — every class draws from the same shared skill and passive pools.

Skills are loot, not unlocks

Active skills and support gems drop from the world and level up by sacrificing gems at fonts — not from character levels. Character XP buys passive points instead. Your build is the skills you find and the supports you socket into them.

Data in, no engine changes

Adding a skill, support, monster, passive, proc, item base, affix, unique, or class means adding an entry under src/data/. The engine already knows how to run it — the same seams a new line reaches are the seams every existing line uses.

Correctness

Harnesses, not vibes

There's no unit-test runner. Correctness is gated by type-checking plus a set of headless harnesses that boot the real engine — so a change to the data can't quietly break the game.

npm run sim

The balance harness — the real engine, headless and deterministic (seeded). Ranks every skill at equal investment, runs the skill × support no-op matrix, prints per-class power curves, and audits the item economy.

npm run perf

The performance harness — boots the real desktop game and samples frame telemetry per procedural biome, gating each against a town control.

npm run genqa

The generation QA sweep — runs level generation across the whole authored tileset/variant matrix over several seeds and asserts the generation invariants.

npm run eventqa

The event-fabric gate — a 168-check invariant sweep over every world-event: persistence, ownership claims, and full parity with the game's own event tab.
